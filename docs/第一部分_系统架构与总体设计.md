# 第一部分：系统架构与总体设计

## 1.1 技术栈选型

本古诗学习平台采用前后端分离的架构设计，技术栈选型遵循轻量化、高效性和可扩展性原则。

### 1.1.1 后端技术栈

**核心框架**：采用 Flask 2.3.3 作为 Web 应用框架。Flask 是 Python 生态中的轻量级框架，具有灵活性高、学习曲线平缓、扩展性强的特点，适合快速构建 RESTful API 服务。

**数据持久化**：使用 SQLAlchemy 2.0.23 作为 ORM（对象关系映射）框架，配合 SQLite 数据库。SQLite 作为嵌入式数据库，无需独立服务器进程，部署简便，适合中小规模应用的原型开发和测试。数据库包含 8 个核心数据表，涵盖用户管理、古诗资源、学习行为追踪、练习系统和 AI 交互记录等功能模块。

**身份认证与安全**：采用 Flask-JWT-Extended 4.5.3 实现基于 JSON Web Token 的无状态身份认证机制，配合 Flask-Bcrypt 1.0.1 对用户密码进行哈希加密存储，确保系统安全性。JWT Token 有效期设置为 24 小时，平衡了用户体验与安全性需求。

**跨域资源共享**：通过 Flask-CORS 4.0.0 处理跨域请求，支持前后端分离部署和多端访问。

### 1.1.2 前端技术栈

**基础技术**：采用原生 HTML5、CSS3 和 JavaScript（Vanilla JS）开发，未引入 React、Vue 等重型前端框架。这一选择基于以下考虑：（1）降低项目复杂度，减少构建工具依赖；（2）提升页面加载速度，优化用户体验；（3）便于快速迭代和功能调试。

**模板引擎**：使用 Flask 内置的 Jinja2 模板引擎进行服务端渲染，实现页面的动态生成和数据绑定。

**交互追踪**：前端集成了鼠标行为追踪、注意力分析和交互事件记录功能，通过 AJAX 异步传输数据至后端，为学习行为分析提供数据支撑。

### 1.1.3 人工智能与语音处理技术栈

**大语言模型**：集成通义千问 Qwen2.5-7B-Instruct 模型，通过 SiliconFlow 平台提供的 API 接口调用。选择 Qwen 模型的核心理由包括：

1. **卓越的中文处理能力**：Qwen 系列模型专门针对中文语料进行优化训练，对古诗词、文言文等传统文化内容具有深刻理解能力，能够准确解析字词含义、诗歌意境和文化背景。

2. **性价比优势**：相比 GPT-4 等商业闭源模型，通过 SiliconFlow 平台调用 Qwen 模型成本显著降低，按需付费模式适合教育应用场景的预算控制。

3. **多语言支持**：系统支持检测和响应中文、英语、西班牙语、法语、德语、日语、韩语等多种语言，满足国际汉语学习者的需求。Qwen 模型在多语言任务上表现优异，能够根据学习者的母语提供个性化解释。

4. **自适应难度调节**：系统根据用户的 HSK 等级（HSK1-HSK6）动态调整 AI 回答的复杂度和字数限制，确保解释内容与学习者的汉语水平相匹配。

**语音技术**：
- **语音合成（TTS）**：采用 edge-tts 6.1.9，基于微软 Edge 浏览器的免费 TTS 服务，支持多语言、多音色，可为古诗生成高质量朗读音频。
- **语音识别（ASR）**：集成 OpenAI Whisper 模型，用于语音输入和发音评估功能。
- **音频处理**：使用 librosa 0.10.1 进行音频特征提取，pydub 0.25.1 处理音频格式转换，为语音训练模块提供技术支撑。

### 1.1.4 系统架构设计

系统采用经典的三层架构模式：

**表示层（Presentation Layer）**：由前端页面和用户交互界面组成，包括古诗列表页、学习详情页、练习系统页和数据仪表板。前端通过 RESTful API 与后端通信，实现数据的异步加载和动态更新。

**业务逻辑层（Business Logic Layer）**：Flask 应用核心，包含 9 个功能模块的 API 路由（认证、古诗管理、学习记录、练习系统、AI 对话、数据仪表板、注意力追踪、行为追踪、语音训练）。该层负责业务规则处理、数据验证和服务调度。

**数据访问层（Data Access Layer）**：通过 SQLAlchemy ORM 封装数据库操作，提供统一的数据访问接口。数据库设计遵循第三范式，合理设置外键关联，确保数据一致性和完整性。

**服务层（Service Layer）**：独立的服务模块，包括 Qwen AI 服务（`qwen_service.py`）和语音训练服务（`voice_training_service.py`），实现与外部 API 的交互和复杂业务逻辑的封装。

### 1.3.2 核心数据表说明

数据库共包含 8 个核心数据表，各表功能如下：

1. **users（用户表）**：存储用户基本信息、学习统计数据和个性化设置（HSK 等级、母语）。
2. **poems（古诗表）**：存储古诗的标题、作者、朝代、内容、翻译、背景、赏析、多媒体资源路径等。
3. **learning_records（学习记录表）**：记录用户对每首古诗的学习行为，包括阅读时长、视频观看时长、音频收听时长、字词查询记录和 AI 交互次数。
4. **exercises（练习题表）**：存储练习题目，包括题型、题干、选项、正确答案和解析。
5. **user_exercise_records（答题记录表）**：记录用户的答题情况，包括用户答案、正误判断、得分和答题用时。
6. **qa_records（问答记录表）**：存储用户与 AI 的问答历史，包括问题、答案、问题类型、用户评分和响应时间。
7. **attention_tracking（注意力追踪表）**：记录用户的鼠标行为数据（移动、点击、滚动）和注意力指标（专注时长、分心次数、参与度评分）。
8. **learning_session 和 interaction_event（学习会话与交互事件表）**：记录每次学习会话的详细信息和细粒度的交互事件，支持学习行为的深度分析。

### 1.3.3 数据关系设计

数据表之间通过外键建立关联关系，形成完整的数据模型：
- 用户（users）与学习记录、答题记录、问答记录、学习会话之间为一对多关系。
- 古诗（poems）与学习记录、练习题、问答记录之间为一对多关系。
- 练习题（exercises）与答题记录之间为一对多关系。
- 学习会话（learning_session）与交互事件（interaction_event）之间为一对多关系。

这种关系设计确保了数据的规范化存储，避免冗余，同时支持高效的关联查询和数据分析。
