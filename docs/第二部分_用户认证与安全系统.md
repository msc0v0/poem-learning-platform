# 第二部分：用户认证与安全系统 (User Management & Security)

## 2.1 安全架构设计

本系统采用多层次的安全防护机制，确保用户数据的私密性和系统访问的安全性。安全架构包括密码加密存储、基于令牌的身份认证、会话管理和访问控制四个核心模块。

## 2.2 密码安全存储机制

### 2.2.1 Bcrypt 哈希算法

系统使用 **bcrypt** 算法对用户密码进行单向哈希加密，这是密码安全存储的行业标准实践。Bcrypt 的核心优势包括：

1. **自适应哈希函数**：基于 Blowfish 加密算法，计算成本可调节，能够有效抵御暴力破解攻击。
2. **自动加盐（Salt）**：每个密码生成唯一的随机盐值，即使两个用户使用相同密码，存储的哈希值也完全不同，防止彩虹表攻击。
3. **不可逆性**：哈希过程不可逆，即使数据库泄露，攻击者也无法还原原始密码。

在用户注册时（`models/user.py`），系统通过 `bcrypt.generate_password_hash()` 方法将明文密码转换为 128 位哈希值存储：

```python
def __init__(self, username, email, password):
    self.username = username
    self.email = email
    self.password_hash = bcrypt.generate_password_hash(password).decode('utf-8')
```

用户登录时，系统使用 `bcrypt.check_password_hash()` 方法验证密码，而非直接比对字符串，确保密码验证的安全性。

### 2.2.2 密码验证流程

登录验证采用恒定时间比较算法，防止时序攻击（Timing Attack）。验证逻辑如下：

```python
def check_password(self, password):
    return bcrypt.check_password_hash(self.password_hash, password)
```

## 2.3 JWT 会话管理机制

### 2.3.1 JWT 令牌结构

系统采用 **JSON Web Token (JWT)** 实现无状态的会话管理。JWT 是一种开放标准（RFC 7519），用于在客户端和服务器之间安全传输用户身份信息。JWT 由三部分组成：

- **Header（头部）**：声明令牌类型和加密算法（如 HS256）
- **Payload（载荷）**：存储用户身份标识（user_id）和令牌有效期等声明
- **Signature（签名）**：使用密钥对头部和载荷进行签名，防止令牌被篡改

### 2.3.2 令牌生成与验证

**令牌生成**：用户成功登录或注册后，系统调用 `create_access_token()` 生成 JWT 令牌，有效期为 24 小时：

```python
def generate_token(self):
    return create_access_token(identity=self.id)
```

**令牌验证**：客户端在后续请求中将 JWT 令牌放入 HTTP 请求头的 `Authorization` 字段（格式：`Bearer <token>`）。服务器通过 `@jwt_required()` 装饰器自动验证令牌的有效性和完整性，并提取用户身份：

```python
@auth_bp.route('/profile', methods=['GET'])
@jwt_required()
def get_profile():
    user_id = get_jwt_identity()  # 从JWT中提取用户ID
    user = User.query.get(user_id)
    return jsonify({'user': user.to_dict()}), 200
```

### 2.3.3 安全优势

JWT 机制的核心优势：

1. **无状态性**：服务器无需存储会话信息，降低服务器内存开销，支持水平扩展。
2. **跨域支持**：适合前后端分离架构和移动端应用。
3. **防篡改**：签名机制确保令牌内容不可被伪造或修改。
4. **自动过期**：令牌内置过期时间，降低令牌泄露风险。

## 2.4 认证流程实现

### 2.4.1 用户注册流程

1. 前端收集用户名、邮箱和密码，进行客户端验证（密码长度≥6位）
2. 发送 POST 请求至 `/api/auth/register`
3. 后端验证数据完整性，检查用户名和邮箱唯一性
4. 使用 bcrypt 加密密码，创建用户记录并存入数据库
5. 生成 JWT 令牌并返回给客户端
6. 客户端将令牌存储在 `localStorage` 中，用于后续请求

### 2.4.2 用户登录流程

1. 前端收集用户名和密码
2. 发送 POST 请求至 `/api/auth/login`
3. 后端查询用户记录，使用 bcrypt 验证密码
4. 验证通过后更新 `last_login` 时间戳
5. 生成 JWT 令牌并返回
6. 客户端存储令牌，更新 UI 显示用户信息

### 2.4.3 受保护资源访问

所有需要身份验证的 API 端点（如学习记录、练习提交、AI 对话）均使用 `@jwt_required()` 装饰器保护。客户端在请求时自动携带 JWT 令牌，服务器验证通过后才允许访问资源，确保用户学习数据的私密性和完整性。

## 2.5 安全增强措施

1. **账户状态管理**：`is_active` 字段支持账户禁用功能，防止异常账户访问。
2. **密码修改验证**：修改密码时需验证当前密码，防止未授权修改。
3. **令牌过期处理**：系统配置统一的令牌过期回调，自动返回 401 错误，提示用户重新登录。
4. **输入验证**：所有 API 端点进行严格的输入验证，防止 SQL 注入和 XSS 攻击。
